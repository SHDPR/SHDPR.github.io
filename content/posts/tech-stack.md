블로그 개발 시 각 기술을 왜 선택했는지, 실제로 써보니 어떤지 — 솔직하게 정리했습니다.

## Next.js — 프레임워크

React 기반의 풀스택 웹 프레임워크입니다.

가장 큰 특징은 **서버 컴포넌트**와 **클라이언트 컴포넌트**를 구분한다는 점입니다. 데이터를 서버에서 불러오는 컴포넌트(위젯, 포스트 목록 등)는 서버에서 렌더링되고, 버튼처럼 사용자 인터랙션이 필요한 컴포넌트만 브라우저에서 실행됩니다. 덕분에 불필요한 JavaScript가 클라이언트로 전달되지 않습니다.

URL 구조는 폴더 구조로 그대로 표현됩니다. `/blog/[slug]` 폴더를 만들면 포스트 페이지가 되고, `/api/views/[slug]`는 조회수 API가 됩니다. 처음엔 낯설었지만 익숙해지면 직관적입니다.

| 장점 | 단점 |
|---|---|
| 서버/클라이언트 분리로 성능 최적화 | 학습 곡선이 있음 |
| 파일 기반 라우팅이 직관적 | 단순한 블로그엔 다소 과한 선택 |
| Vercel과 완벽하게 통합 | 빌드 시간이 상대적으로 김 |
| TypeScript 기본 지원 | App Router 관련 문서가 아직 부족 |

## Tailwind CSS — 스타일링

클래스 이름만으로 디자인을 완성하는 CSS 프레임워크입니다.
`text-sm`, `font-bold`, `rounded-xl` 같은 유틸리티 클래스를 조합해서 별도의 CSS 파일 없이 UI를 만들 수 있습니다.

이 블로그의 다크/라이트 테마는 CSS 커스텀 변수로 구현했습니다.

```css
:root, [data-theme="dark"] {
  --bg: #1a1a1a;
  --accent-1: #ff1e63;
}
[data-theme="light"] {
  --bg: #ffffff;
}
```

`data-theme` 속성만 바꾸면 변수가 일괄 교체되면서 전체 색상이 바뀝니다. 테마 전환 버튼은 이 속성을 토글하는 것뿐입니다.

다만 v4에서 커스텀 폰트를 CSS 클래스로 적용하면 Tailwind의 cascade가 덮어써서 반영이 안 됩니다. 따라서 폰트는 인라인 `style` 속성으로 직접 적용해야 했습니다.

| 장점 | 단점 |
|---|---|
| 클래스만으로 빠른 UI 구현 | 클래스가 길어지면 가독성이 떨어짐 |
| CSS 파일 관리 부담 없음 | v4는 아직 변경사항이 잦음 |
| 반응형 처리가 직관적 | 커스텀 폰트 적용 시 주의 필요 |

## 폰트

영어와 한국어가 공존하는 블로그다 보니 폰트를 두 가지로 나눴습니다.

| 역할 | 폰트 |
|---|---|
| 포스트 제목 / 본문 (영문) | Playfair Display |
| 블로그 로고 | DM Serif Display |
| 히어로 제목 | Syne |
| UI 전반 (영문) | Geist Sans |
| 한국어 전반 | Do Hyeon |

포스트 본문에는 Playfair Display를 골랐습니다. 세리프 계열 폰트 12종을 미리보기 페이지로 만들어서 직접 비교한 뒤 선택했습니다. 세련된 느낌과 가독성 모두 만족스러웠습니다.

## Upstash Redis — 데이터베이스

포스트 조회수와 방문자 추이를 저장하는 데 사용합니다.

서버리스 환경에서 일반 Redis를 쓰면 문제가 생깁니다. 함수 실행마다 연결을 새로 맺어야 하는데, TCP 연결은 무겁습니다. Upstash는 HTTP API로 동작해서 이 문제가 없습니다.

실제로 사용하는 방식은 단순합니다.
- 포스트를 열면 해당 slug의 조회수가 1 증가합니다 (`INCR post:views:{slug}`)
- 하루에 한 번, 방문자 수가 집계됩니다 (`INCR blog:daily:visits:YYYY-MM-DD`)
- 사이드바의 스파크라인은 14일치 데이터를 꺾은선 SVG로 시각화합니다

같은 사람이 새로고침할 때마다 조회수가 오르는 걸 막기 위해, 브라우저 localStorage에 방문 여부를 하루 단위로 기록합니다.

| 장점 | 단점 |
|---|---|
| 서버리스 환경과 궁합이 좋음 | 무료 플랜에 요청 수 제한 있음 |
| HTTP API라 연결 부담 없음 | 복잡한 쿼리는 불가 |
| 응답이 빠름 | 영속성보다 캐시에 최적화된 구조 |

## Giscus — 댓글

GitHub Discussions를 댓글 저장소로 활용하는 오픈소스 위젯입니다.

별도 서버나 DB가 필요 없고, 댓글을 달려면 GitHub 계정이 필요합니다. 기술 블로그 특성상 방문자 대부분이 GitHub를 쓴다고 판단해서 선택했습니다. 실제로 스팸은 거의 없습니다.

포스트 페이지 하단에 자동으로 삽입되며, 다크/라이트 테마와도 연동됩니다.

| 장점 | 단점 |
|---|---|
| 서버/DB 불필요, 완전 무료 | GitHub 계정이 있어야 댓글 가능 |
| 스팸이 사실상 없음 | 비개발자 방문자에겐 진입 장벽 |
| 테마 연동 지원 | GitHub 서비스에 의존적 |

---

## Fuse.js — 검색

클라이언트 사이드 퍼지 검색 라이브러리입니다.

헤더의 검색 버튼을 클릭하면 모달이 열리고, 포스트 제목과 설명을 대상으로 실시간 검색합니다. 서버에 요청을 보내지 않기 때문에 응답이 즉각적입니다. 오타가 있어도 비슷한 결과를 찾아줍니다.

포스트 수가 적은 지금은 이 방식이 최선입니다. 나중에 포스트가 수백 개가 되면 서버 사이드 검색을 고려할 수도 있습니다.

---

## Vercel — 배포

Next.js를 만든 회사의 배포 플랫폼입니다. GitHub 저장소를 연결하면 `git push`만으로 자동 빌드 및 배포가 됩니다.

이 블로그에서 Vercel을 활용하는 방식이 하나 더 있습니다. `x-vercel-ip-country` 헤더를 통해 접속자의 국가를 감지해서 기본 언어를 자동으로 설정합니다. 한국에서 접속하면 한국어, 해외에서 접속하면 영어가 기본이 됩니다.

| 장점 | 단점 |
|---|---|
| Next.js와 완벽한 통합 | 트래픽이 많으면 유료 전환 필요 |
| git push만으로 자동 배포 | 벤더 종속성 |
| 지역 감지 등 부가 기능 | 빌드 시간 초과 시 제한 있음 |
| 개인 블로그는 무료 플랜으로 충분 | |

## 국제화 (i18n) — 직접 구현

별도 라이브러리 없이 만들었습니다.

언어는 세 단계로 결정됩니다. 사용자가 직접 토글을 누른 경우 쿠키에 저장된 선택이 최우선입니다. 쿠키가 없으면 Vercel의 지역 감지로 결정하고, 그것도 없으면 한국어가 기본입니다.

포스트는 `{slug}.md`(한국어)와 `{slug}.en.md`(영어) 두 파일로 관리합니다. 언어를 전환하면 서버에서 해당 파일을 읽어 반환합니다. 실시간 번역이 아닌 정적 파일 방식이라 빠르고 안정적입니다.

## 구현된 기능 전체 목록

| 기능 | 구현 방식 |
|---|---|
| 다크/라이트 테마 | CSS 커스텀 변수, 시스템 설정 기본값 |
| 언어 전환 (KO/EN) | 쿠키 + Vercel 지역 감지 |
| 포스트 조회수 | Upstash Redis INCR |
| 방문자 추이 스파크라인 | Redis 14일 데이터 + SVG |
| 클라이언트 검색 | Fuse.js 퍼지 검색 |
| 댓글 | Giscus (GitHub Discussions) |
| 인기 포스트 위젯 | Redis 조회수 기준 정렬 |
| 카테고리 / 태그 | 마크다운 프론트매터 |
| Sitemap | `/sitemap.xml` 자동 생성 |
| Google Search Console | 소유권 인증 + sitemap 제출 |
| 이중 언어 포스트 | `.md` + `.en.md` 정적 파일 |
| SEO 메타태그 | Next.js metadata API |

---

이 스택의 공통점은 하나입니다. **서버를 직접 운영하지 않아도 된다**는 것.

`git push` 하나로 전 세계에 배포되고, Redis와 Vercel이 나머지를 처리합니다.
개인 블로그를 운영하기에 충분하고, 비용은 공짜입니다.
